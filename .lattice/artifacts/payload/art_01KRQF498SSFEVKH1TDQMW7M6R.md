# Plan Review: C11-27 — Split `c11Tests` into pure-logic vs host-required

## 1. Verdict

**PASS (with required corrections)** — The plan is thorough, internally consistent, and well-specified down to the exact YAML and pbxproj operations. Two pre-implementation fixes are required (one factual classification error in §1, one methodology footgun in §1.5), but neither blocks the overall approach. Implementation can proceed once those are corrected in the plan; both fall inside the implementer's first commit and don't require a second planning pass.

## 2. Summary

Reviewed the C11-27 plan to split `c11Tests` into a host-required bundle (`c11Tests`, scheme `c11-unit`) and a new logic-only bundle (`c11LogicTests`, scheme `c11-logic`) so 80 tests can run locally without spawning a frozen `c11 DEV.app` host. The plan is unusually well-scoped: spike-first linkage decision (Strategy B with Strategy A fallback), explicit per-file CI workflow diffs, verbatim CLAUDE.md rewrite, and a comprehensive risk register. **Key concern:** §1's per-file PURE/HOST verdict has at least one verifiably wrong classification (`FlashColorParsingTests` is marked PROMOTE but its body uses NSColor methods throughout), which calls the grep-only audit's reliability into question and suggests other PROMOTEs deserve a second look before the implementer commits the import-drop change.

## 3. Issues

**[MAJOR] §1 — `FlashColorParsingTests` PROMOTE classification is factually wrong**
The plan's row says: "`NSColor` appears in a doc-comment only … PROMOTE; drop `import AppKit` (line 2); body uses no AppKit symbol." This is incorrect. The test body calls `color.usingColorSpace(.sRGB)`, `srgb.redComponent`, `srgb.greenComponent`, `srgb.blueComponent`, `srgb.alphaComponent` across 7+ test functions (verified against `c11Tests/FlashColorParsingTests.swift:14–77`). `FlashAppearance.parseHex(...)` returns `NSColor?`, and every accessor used downstream is an NSColor API — none of those properties exist on `CGColor`. Dropping `import AppKit` here will fail at compile time. The plan's PROMOTE-recovery policy ("if any PROMOTE file fails to compile after the import drop, revert and demote to HOST") will catch this at implementer workflow step 3, so the project will not ship broken — but the fact that a "re-audit each HOST file with ≤2 hits for cosmetic vs. real usage" pass missed an obvious case is the real signal.
**Recommendation:** Demote `FlashColorParsingTests` from PROMOTE → HOST in §1 *before* implementation starts. Then re-audit the remaining 6 PROMOTE files specifically by checking method calls on AppKit-typed return values, not just direct symbol references. The audit method documented in §1.5 (the PascalCase grep) won't catch this pattern either — see next issue.

**[MAJOR] §1.5 — Transitive-dependency audit's grep methodology cannot detect AppKit usage that comes through return types**
The audit script greps test files for `[A-Z][A-Za-z0-9_]+` identifiers and looks up where each is declared in `Sources/`. This misses the most common AppKit leak: a test calls `FlashAppearance.parseHex(...)` (a c11 type — clean) and chains `.usingColorSpace(.sRGB)?.redComponent` on it (NSColor methods — never named explicitly). The audit's output would say "FlashColorParsingTests references FlashAppearance, which lives in `Sources/Theme/FlashAppearance.swift`" — true but insufficient, because `FlashAppearance.parseHex` returns NSColor, which pulls AppKit transitively whether or not `Sources/Theme/FlashAppearance.swift` imports AppKit directly.
**Recommendation:** Drop §1.5's pretense that grep can prove hostlessness. Replace the audit with: "The compiler is the audit." Specifically, after the spike commit and bulk move, the success criterion is that `xcodebuild build -scheme c11-logic` succeeds. Files that fail to compile get reverted to HOST and the script re-run. This is faster than maintaining a checked-in `.lattice/plans/c11-27-deps.txt` whose accuracy nobody can independently verify, and it avoids the trap of false confidence the grep audit creates. If a checked-in dependency artifact is desired for the PR review, generate it from `swiftc -dump-ast` or `xcodebuild -showBuildSettings`-derived module dependencies, not PascalCase regex.

**[MAJOR] §2.2 — Spike candidates do not exercise the failure mode being tested**
The spike runs `MailboxIOTests` and `StdinHandlerFormattingTests` — both genuinely clean files in subsystems (Mailbox, CLI runtime) that don't touch AppKit even transitively. A green spike here proves Strategy B works *for files that have no AppKit pressure at all*. It does **not** prove Strategy B works for the Theme/Workspace tests that make up the bulk of the migration, where transitive AppKit pull is the real concern. The spike will green-light bulk roll-out, then bulk roll-out will surface the AppKit-via-return-type issues file by file.
**Recommendation:** Add one more spike candidate from the Theme cluster — e.g., `ThemeRegistryTests` (clean per the audit) plus one PROMOTE-candidate like `ThemeResolvedSnapshotArtifactTests` (single `import AppKit`; body claims to only read enums + write JSON). If both pass after dropping `import AppKit`, confidence in the bulk move increases substantially. If `ThemeResolvedSnapshotArtifactTests` fails like `FlashColorParsingTests` would, the implementer learns at spike time, not bulk-move time.

**[MAJOR] §2.1 Strategy A scope risk — "may demote a significant fraction" is under-quantified**
The plan acknowledges that Strategy A's dual-compile may force PURE→HOST demotions when a source file transitively imports AppKit. But it doesn't define a threshold for when Strategy A stops being worth it. If `Sources/Theme/*.swift` imports AppKit (likely, given NSColor usage), then *every* Theme test that touches a Theme type would demote — that's most of the Theme cluster (~15 files). Combined with Workspace tests that touch types whose sources import SwiftUI (which most do), Strategy A could end up moving only the ~25 truly self-contained tests (Mailbox, parsers, CLI runtime) and leaving the rest in HOST. That's a much smaller win than the plan's "80 logic tests" framing implies.
**Recommendation:** Before locking in Strategy A from a spike pass, require the implementer to run a *full* dependency check (compile the entire candidate set under Strategy A and count successful inclusions) and report the number to a step in the workflow. If the post-demotion count falls below, say, 50 files, escalate to Strategy B even if Strategy A's spike passed. The §8 acceptance criteria should add a minimum-coverage gate: e.g., "c11LogicTests contains ≥50 test files" — otherwise C11-27 isn't moving the needle enough to justify the new target.

**[MINOR] §2.2 — Spike depends on `c11-logic` scheme that doesn't exist yet**
Step 1 of §2.2 says `xcodebuild build -scheme c11-logic` but the scheme is created in §3 step 7. The plan's implementer-workflow §3 ordering puts spike (step 4) before scheme creation (step 7). The spike either needs (a) inline scheme creation as part of the spike commit, or (b) the spike target built via `-target c11LogicTests` instead of via scheme. Either works; the plan should pick one and update §2.2/§3 to match.
**Recommendation:** Add the scheme creation to step 4 ("Spike Strategy B") so the spike commit is self-contained — script-add target + create scheme + move 2 files + verify pgrep. Then steps 5–9 just extend membership and update CI; no scheme re-work needed mid-flow.

**[MINOR] §2.2 — `pgrep` acceptance check is necessary but not sufficient**
The spike gate is "during the run, `pgrep -fl 'c11 DEV.app'` returns empty." Good. But Strategy B sets `BUNDLE_LOADER = "$(BUILT_PRODUCTS_DIR)/c11.app/Contents/MacOS/c11"` (note: `c11.app`, not `c11 DEV.app`). If Strategy B silently launches the *release* c11.app (which the build produces alongside DEV.app in some configurations), the `pgrep` for `c11 DEV.app` would return empty — false green — and the operator would get bitten by a different launching app later.
**Recommendation:** Broaden the pgrep pattern: `pgrep -fl '/c11(\.app| DEV\.app)/Contents/MacOS/c11'` (or run both invocations: one for each app name). Alternatively, use `lsof` against the xctest PID to confirm no `c11*.app` binary is loaded. The §8 acceptance criteria should reflect the broader check.

**[MINOR] §4.3 / §4.4 — "Leave unchanged" claim should verify scheme references, not just step names**
The plan dismisses `ci-macos-compat.yml` and `test-e2e.yml` because the relevant line is a `-resolvePackageDependencies` step. True, but if either workflow elsewhere references `c11-ci` (which now gains `c11LogicTests` as a TestableReference per §4.5), they'll silently start running the new target too — possibly desirable, possibly not. Worth a one-line grep to confirm.
**Recommendation:** Add to the §4.3/§4.4 verification: `grep -nE 'c11-(ci|unit|logic)|c11Tests|c11LogicTests' .github/workflows/ci-macos-compat.yml .github/workflows/test-e2e.yml`. If only the `-resolvePackageDependencies` step matches, the "leave unchanged" verdict holds. If anything else matches, evaluate explicitly.

**[MINOR] §4.x wall-time projection conflates test phase with total invocation**
The "~5–10 s" projection for `c11-logic test` is test-phase time only. `xcodebuild` itself has ~10–15 s of inherent overhead (scheme resolution, package check, project parse, build-phase scheduling) even on a fully cached build. Total wall time for a `c11-logic test` invocation will be closer to 20–25 s. That's still a major win over today's ~35 s plus the operator-frustration cost, but the §5 CLAUDE.md text says "Expected wall time on a warm cache: under 20 seconds" — set the expectation at "30 seconds, dominated by xcodebuild overhead rather than test execution" so the operator doesn't feel cheated when the first run takes longer than the projection.
**Recommendation:** Soften §5's wall-time claim. Document the xcodebuild overhead floor so the operator's mental model matches reality.

**[MINOR] §3 — Ruby script's `STRATEGY_A_SOURCES` placeholder will burn an implementer**
The script has a stub `STRATEGY_A_SOURCES = %w[ Sources/Mailbox/MailboxEnvelope.swift Sources/Mailbox/MailboxDispatcher.swift # ... regenerated from the §1.5 audit ]`. An implementer running `STRATEGY=A ruby scripts/c11-27-split-tests.rb` without first regenerating that list will create a target that compiles only the Mailbox subset — a confusing partial result. The script should either fail loudly when the list looks like a stub (e.g., refuse to run if STRATEGY_A_SOURCES has fewer than 20 entries, or read the list from `.lattice/plans/c11-27-deps.txt` and abort if the file is missing), or be re-architected so the list cannot be stale.
**Recommendation:** Have the script `abort` if `ENV['STRATEGY'] == 'A'` and `.lattice/plans/c11-27-deps.txt` doesn't exist or is empty. Then `STRATEGY_A_SOURCES` becomes `File.readlines('.lattice/plans/c11-27-deps.txt').map(&:chomp).reject(&:empty?)`. Single source of truth, no stale-stub trap.

**[MINOR] §6 — Risk register missing "PROMOTE classification error" row**
Given the FlashColor finding above, this is now a known risk class, not a hypothetical. The register lists "PURE-classified file pulls AppKit transitively" but not "PROMOTE-classified file's body uses an AppKit type the grep missed." Worth surfacing as its own row so the implementer knows to expect at least one or two more demotions during workflow steps 2–3.
**Recommendation:** Add row: *"PROMOTE classification missed an AppKit type accessed through a c11 return value. Likelihood: Medium (1/7 confirmed). Impact: Low (caught at step 3 build). Mitigation: implementer demotes file in script before re-run."*

**[MINOR] §1 table totals — 73 PURE + 7 PROMOTE + 21 HOST = 101 — table rows don't match the tally**
Counting the rows in §1 yields 93 entries (visible in the prompt). 101 files exist (verified: `ls c11Tests/*.swift | wc -l` → 101). 8 test files are unaccounted for. They may be intentionally omitted (test helpers, base classes that XCTest doesn't run), but the plan doesn't say so. A reviewer can't tell whether the 8 missing files are pure or host.
**Recommendation:** Either include the missing 8 rows (with a "TEST HELPER, no XCTestCase subclass" verdict where applicable) or add a paragraph naming the omissions and why they don't need classification. The implementer needs to know whether to touch those files' target membership.

## 4. Positive Observations

- **Spike-first protocol with concrete go/no-go criteria (§2.2).** Most plans say "we'll figure out linkage during implementation." This one commits to two candidate files, an explicit acceptance gate (`pgrep` during test run), and a fallback ladder. That's the right shape — Strategy B is cheap to try, falls back to A cleanly, and Strategy C is explicitly out of scope.
- **Verbatim CLAUDE.md rewrite (§5) instead of a description of the edit.** The implementer doesn't have to re-derive the prose. The new text correctly identifies the lie in the old policy ("safe (no app launch)") and replaces it with a precise rule. The "Rule of thumb" paragraph at the end is exactly the kind of guidance that helps future agents navigate the two schemes correctly.
- **Per-file CI workflow diffs (§4.1–4.6) with exact YAML.** The mailbox-parity selector update (§4.2) catches a silent-pass mode — without it, the 10 `-only-testing:c11Tests/X` selectors would match zero tests post-split and CI would go green over nothing. That's the kind of failure mode plan-review is meant to surface; the plan caught it itself.
- **Acceptance criteria are mechanical (§8).** "TEST_HOST is empty for c11-logic per `xcodebuild -showBuildSettings`" is testable; "no DEV.app launches" plus the `pgrep` command is testable; "test phase under 20 s" is measurable. Compare with the ticket's looser "<5 seconds" and "no frozen DEV.app window" — this is a real improvement.
- **`c11-ci` coverage regression caught (§4.5).** Without §4.5, the dedicated CI scheme would silently drop 80 tests. Calling this out explicitly with the exact XML stanza to add is the kind of detail that prevents a "we shipped it but lost coverage" PR-review-time discovery.
- **MOVE-don't-duplicate policy (§2.4).** Avoids the double-execution-of-mutable-state class of bug, with a clear reason given. The decision is locked, not deferred.
- **Risk register (§6) is actually mitigation-oriented.** Each row names a specific countermeasure rather than just acknowledging the risk exists. The "PR #164 conflict" risk noted in the ticket is correctly retired ("Was-medium, now mitigated") — the plan tracks what's been resolved, not just what remains.
- **Provenance, worktree, branch name, and rebased SHA all named upfront.** A future agent picking this up after a context loss can locate the work in one read.
