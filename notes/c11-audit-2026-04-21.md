Date: 2026-04-21

Auditor context: running inside C11 (`CMUX_SHELL_INTEGRATION=1`)

## Executive Summary

This audit found no single obvious "ship stopper" such as a missing target or a broken project graph. The app, CLI, remote daemon, browser, terminal, markdown, socket, and skill surfaces are all present and broadly coherent.

The main risks are contract drift and operational complexity:

1. **High: `cmux` compatibility does not reliably resolve to the same binary as `c11`.** In this live C11 session, `c11` resolves to the active bundled CLI, but `cmux` resolves first to `/opt/homebrew/bin/cmux`, which dispatches to an upstream `cmux 0.62.0` app. Several documented C11 commands then fail.
2. **High: the remote daemon CLI exposes only a small subset of the local C11 CLI.** Remote shells get a `$HOME/.cmux/bin/c11`/`cmux` wrapper, but commands such as `set-agent`, `set-metadata`, `set-status`, `set-progress`, `markdown`, and titlebar commands are absent from the remote command table.
3. **Medium-high: socket command handling still has main-thread synchronization in hot telemetry paths.** The repo's own policy calls out this risk; static inspection shows `set_status`, `set_progress`, `log`, list/clear commands, and every v2 call still hit synchronous main-thread sections in places.
4. **Medium-high: first-party Swift is concentrated into very large files.** `TerminalController.swift`, `CLI/c11.swift`, `ContentView.swift`, `AppDelegate.swift`, `Workspace.swift`, and `BrowserPanel.swift` hold too many domains each. This is increasing review cost and regression risk.
5. **Medium: validation governance is inconsistent.** Local instructions say not to run tests locally except limited safe cases, CI intentionally builds without xctest in the main workflow, while the PR checklist still asks authors to confirm local testing.
6. **Medium: security-sensitive capabilities are broad and should have a maintained threat model.** This may be appropriate for a terminal/browser host, but it deserves explicit review because the app has browser handling, Apple Events, camera/mic entitlements, unsigned executable memory/JIT, and a scriptable socket.

The strongest immediate fixes are to harden CLI resolution, bring the remote CLI wrapper up to the documented agent contract, and remove synchronous main-thread work from telemetry-style socket commands.

## Findings

### 1. High: `cmux` Compatibility Alias Is Not Reliable In Live C11 Shells

The repo and skill state that the primary binary is `c11` and `cmux` is a compatibility alias that dispatches to the same binary. That is not true in this live C11 session.

Evidence:

- `c11` resolves to the active bundled dev CLI:
  - `/Users/atin/Library/Developer/Xcode/DerivedData/c11-rename-c11/Build/Products/Debug/c11 DEV rename-c11.app/Contents/Resources/bin/c11`
  - `c11 --version` returns `cmux 0.37.0 (87)`.
- `cmux` resolves first to `/opt/homebrew/bin/cmux`.
- `/opt/homebrew/bin/cmux --version` returns `cmux 0.62.0 (75) [c9b775618]`.
- The active bundled CLI is later in PATH at entry 17, after Homebrew.
- `cmux set-agent`, `cmux set-metadata`, `cmux get-metadata`, and `cmux clear-metadata` fail with `Unknown command` through the first `cmux` on PATH.
- The same operations work through `$CMUX_BUNDLED_CLI_PATH`.

Relevant source/docs:

- `skills/c11/SKILL.md` says the binary is `c11`, and `cmux` is a compat alias resolving to the same binary.
- `CLAUDE.md:45` says agents that read the C11 skill learn to call `c11 set-metadata` / `c11 set-status`, and that `cmux` is a compat alias dispatching to the same binary.
- `Resources/welcome.md:49-58` still tells users "the binary is `~/.local/bin/cmux`" and shows only `cmux` examples.
- `Resources/shell-integration/cmux-zsh-integration.zsh:708-723` attempts to move the app `Resources/bin` directory to the front of PATH once at first prompt.

Why it matters:

- Agent behavior depends on reliable CLI discovery.
- The welcome surface teaches `cmux`, not `c11`, so a user or agent can follow in-app instructions and hit the wrong product.
- Wrapper scripts such as `Resources/bin/claude` are robust because they use their sibling `cmux`, but ad hoc shell usage is not.
- This is especially confusing because the wrong binary is not merely older. It is a different upstream release line with a different command surface.

Suggested remediation:

1. Prefer documenting and emitting `c11` everywhere user-facing unless there is a deliberate compatibility reason.
2. Make shell integration repair PATH more aggressively, or at every prompt, when `$GHOSTTY_BIN_DIR` and `$CMUX_BUNDLED_CLI_PATH` disagree with `command -v cmux`.
3. Add a startup health check visible in debug/status surfaces:
   - `command -v c11`
   - `command -v cmux`
   - `c11 --version`
   - `cmux --version`
   - whether both point to the active app bundle.
4. Update `Resources/welcome.md` to use `c11` as the primary command and only mention `cmux` as legacy compatibility.
5. Consider making the app-provided `cmux` wrapper detect a live `CMUX_BUNDLED_CLI_PATH` and dispatch to it before Homebrew/upstream app paths.

### 2. High: Remote CLI Does Not Match The Agent Contract

Remote workspaces install `$HOME/.cmux/bin/c11` and `$HOME/.cmux/bin/cmux` wrappers, but those wrappers execute `c11d-remote-current`, whose CLI table is much smaller than the local CLI.

Evidence:

- `Sources/Workspace.swift:4253-4279` creates a wrapper that execs `$HOME/.cmux/bin/c11d-remote-current`.
- `Sources/Workspace.swift:4282-4295` installs that wrapper as `$HOME/.cmux/bin/c11` and symlinks `$HOME/.cmux/bin/cmux` to it.
- `daemon/remote/cmd/c11d-remote/cli.go:50-78` exposes only:
  - `ping`
  - window/workspace/pane/surface basics
  - `send`, `send-key`
  - `notify`
  - `refresh-surfaces`
- The remote table does not include documented local commands such as:
  - `set-agent`
  - `set-metadata`
  - `get-metadata`
  - `clear-metadata`
  - `set-status`
  - `set-progress`
  - `log`
  - `markdown open`
  - `set-title`
  - `set-description`
  - `get-titlebar-state`

There is an `rpc` passthrough in the remote CLI, and tests cover that path. `tests_v2/test_ssh_remote_cli_relay.py:310-354` verifies `ping`, `list-workspaces`, `new-window`, and `rpc system.capabilities`. It does not verify that the documented skill commands work from a remote agent shell.

Why it matters:

- Remote sessions are where the agent contract matters most.
- A remote agent can have `$HOME/.cmux/bin/c11` on PATH and still fail to run the very commands the skill teaches.
- This breaks the "agent writes telemetry and identity into C11" loop in remote workspaces.

Suggested remediation:

1. Add remote command specs for agent lifecycle and sidebar/title metadata:
   - `set-agent`
   - `set-metadata`
   - `get-metadata`
   - `clear-metadata`
   - `set-status`
   - `clear-status`
   - `set-progress`
   - `clear-progress`
   - `log`
   - `set-title`
   - `set-description`
2. Add a remote parity test that runs these commands through `$HOME/.cmux/bin/c11` over the relay, not only via `rpc`.
3. If the remote CLI intentionally remains small, document that limitation in the skill and welcome docs. Right now the installed command name suggests full parity.

### 3. Medium-high: Hot Socket Telemetry Still Blocks On Main Thread

The repo policy in `CLAUDE.md` says to avoid `DispatchQueue.main.sync` for high-frequency telemetry commands such as status/progress/log metadata updates. The implementation has improved in some explicit-scope paths, but synchronous main-thread work remains in the command dispatcher and telemetry code.

Evidence:

- Every v2 JSON-RPC command calls `v2MainSync { self.v2RefreshKnownRefs() }` before dispatch (`Sources/TerminalController.swift:2025`).
- `v2MainSync` uses `DispatchQueue.main.sync` when off-main (`Sources/TerminalController.swift:3222-3226`).
- `v2ResolveSurfaceForMetadata` says it "Runs off-main", but the implementation calls `v2MainSync` (`Sources/TerminalController.swift:6368-6391`).
- Legacy sidebar mutations resolve tabs synchronously:
  - `resolveTabIdForSidebarMutation` uses `DispatchQueue.main.sync` (`Sources/TerminalController.swift:15559-15568`).
  - `clear_status`, `list_status`, metadata block list/clear, `log`, `list_log`, `set_progress`, and `clear_progress` use main sync (`Sources/TerminalController.swift:15749-16084`).
- The shell integrations call these paths frequently:
  - `Resources/shell-integration/cmux-zsh-integration.zsh` reports tty, shell state, ports, git branch, PR, pwd, and agent kicks around prompt/preexec hooks.
  - `Resources/shell-integration/cmux-bash-integration.bash` has parallel behavior.

Why it matters:

- Agents and shell integration can send telemetry during every prompt transition and command run.
- If the AppKit main thread is busy rendering browser/terminal UI, socket writers can block behind UI work.
- If any raw `DispatchQueue.main.sync` path is ever reached from main, it can deadlock. `v2MainSync` protects against that, but raw syncs are still present in legacy paths.

This is not all equally risky. Some commands are explicit UI mutations and must touch the main thread. The problem is that command parsing, target resolution, and return-path behavior are not consistently separated from the main-actor mutation.

Suggested remediation:

1. Introduce a dedicated telemetry command path:
   - parse and validate off-main,
   - resolve explicit workspace/surface IDs without main sync when possible,
   - enqueue the UI mutation asynchronously,
   - return `OK` after enqueue, not after main-thread completion.
2. Keep synchronous reads only for user-invoked read commands where the caller needs a current answer.
3. Change misleading comments such as the `v2ResolveSurfaceForMetadata` "Runs off-main" comment.
4. Add a regression test or debug harness that floods `set-status`, `set-progress`, `report_pwd`, `report_shell_state`, and `ports_kick` while the UI is busy.

### 4. Medium-high: Very Large Core Files Are Slowing Safe Change

The codebase has several large files that mix multiple domains:

- `Sources/TerminalController.swift` combines socket listener lifecycle, v1 protocol, v2 JSON-RPC, browser automation, notifications, debug/test commands, sidebar telemetry, security policy, and focus policy.
- `CLI/c11.swift` combines argument parsing, socket client behavior, command help, remote SSH setup, browser automation, skill install, Claude hooks, release/download helpers, and output formatting.
- `Sources/AppDelegate.swift`, `Sources/ContentView.swift`, and `Sources/Workspace.swift` combine windowing, UI state, remote workspace lifecycle, sidebar state, and command-facing helpers.

Why it matters:

- Reviews become grep-driven rather than architecture-driven.
- Cross-domain regressions become more likely because unrelated changes share the same files.
- Conflict probability increases when multiple agents or humans work in parallel.
- Swift compile and type-check behavior can degrade on very large SwiftUI/AppKit files.

Suggested remediation:

1. Split `TerminalController.swift` by command family:
   - listener/security/auth
   - v1 command router
   - v2 JSON-RPC router
   - sidebar telemetry
   - browser automation
   - debug/test-only commands
   - metadata/titlebar commands
2. Split `CLI/c11.swift` by command family with a shared parser/client layer.
3. Add a lightweight "module ownership" comment or doc to explain where new socket/CLI commands should land.
4. Set a local size budget for new files or command families. Even a soft "do not add more to TerminalController unless extracting too" rule would help.

### 5. Medium: Validation Policy And CI Coverage Are In Tension

There are many tests and some good targeted workflows, but the guidance is inconsistent.

Evidence:

- `CLAUDE.md:82` says `xcodebuild -scheme c11-unit` is safe but prefer CI.
- `CLAUDE.md:83` says Python socket tests require a tagged running app/socket.
- `.github/workflows/ci.yml:74-79` explicitly says the main macOS build does not run xctest.
- `.github/workflows/ci-macos-compat.yml` and `.github/workflows/test-e2e.yml` contain more test execution, but they are not the same as the primary PR build path.
- `.github/pull_request_template.md:28` still asks contributors to check "I tested the change locally".
- Docs under `docs/` include older references saying never run local `xcodebuild test`, even for `cmux-unit`.

Why it matters:

- Contributors and agents do not have a single source of truth for what validation is expected before PR.
- Behavior changes can land with tests added but not run in the common CI path.
- The project already has a large socket and UI surface where regressions often need explicit harness selection.

Suggested remediation:

1. Define a validation matrix in one canonical doc:
   - pure Swift unit
   - AppKit hosting-view unit
   - Python socket tests
   - remote daemon tests
   - e2e/UI tests
   - browser automation tests
2. Update the PR template to say exactly what counts as acceptable validation for each class of change.
3. If `c11-unit` is considered safe, consider running it in a protected CI workflow for PRs from trusted branches.
4. If local xctest is discouraged, remove local-testing language that implies every author must run it.

### 6. Medium: Security-Sensitive Surface Needs An Explicit Threat Model

The app is a terminal, browser, scripting, and agent-hosting environment, so broad capabilities may be intentional. Still, the security posture should be reviewed as a first-class artifact.

Evidence:

- `c11.entitlements` grants:
  - disable library validation
  - unsigned executable memory
  - JIT
  - camera
  - audio input
  - Apple Events
- `Resources/Info.plist` registers `http` and `https` URL schemes with default handler rank.
- `Resources/Info.plist` enables `NSAllowsArbitraryLoadsInWebContent`.
- Socket control has several modes including `allowAll`; default appears to be `cmuxOnly`, and non-`allowAll` modes use `0600` socket permissions, which is good.

Why it matters:

- This app hosts untrusted web content and terminal workloads.
- It exposes local automation and socket control primitives.
- Camera/mic/Apple Events/JIT entitlements deserve an explicit "why this is required" document, even if each is individually justified.

Suggested remediation:

1. Add `docs/security-threat-model.md` with:
   - trusted/untrusted boundaries,
   - socket auth model,
   - browser content boundary,
   - URL handler behavior,
   - camera/mic behavior,
   - Apple Events usage,
   - JIT/unsigned executable memory rationale.
2. Add a short release checklist item for entitlement and Info.plist changes.
3. Add tests around socket mode defaults and `allowAll` file permissions if not already covered.

### 7. Medium: Brand And Command Naming Drift Remains

The project is in a `cmux` to `c11` transition. Some compatibility is deliberate, but there are still user-facing places where old names are primary.

Examples:

- `Resources/welcome.md:49-58` teaches `cmux` as the main command and says the binary is `~/.local/bin/cmux`.
- `.github/ISSUE_TEMPLATE/config.yml` links to upstream `manaflow-ai/cmux` discussions.
- Issue template strings still use `c11mux`/`cmux` language in several places.
- `Resources/c11.sdef` is still titled "cmux Scripting Dictionary".
- `Resources/c11-themes/README.md` says "c11mux themes".
- `Resources/InfoPlist.xcstrings` still contains "New c11mux Workspace Here" and "New c11mux Window Here".
- `Resources/bin/claude` comments still describe "cmux/c11mux" runtime behavior.

Why it matters:

- Some of this is harmless lineage/compatibility context.
- Some of it directly affects user behavior, especially command examples.
- During an active binary rename, docs drift can turn into runtime mistakes, as seen in the live `cmux` resolution finding.

Suggested remediation:

1. Classify every remaining `cmux`/`c11mux` mention into:
   - intentional lineage,
   - compatibility alias,
   - stale user-facing docs,
   - stale internal variable/file naming.
2. Fix stale user-facing docs first.
3. Keep internal compatibility names only where migration requires them.

### 8. Medium-low: Localization Policy Is Mostly Followed But Not Complete

Most production UI strings use `String(localized:defaultValue:)`, especially in newer agent-skills surfaces. A scan for bare SwiftUI strings found many debug-window strings, which may be acceptable if debug-only. One production-looking example remains:

- `Sources/Panels/BrowserPanelView.swift:757` uses `.accessibilityLabel("Browser omnibar")`.

There are also many debug UI strings in `Sources/c11App.swift` such as "Debug Window Controls", "Copy Config", "Reset", and similar. If those debug windows ship in debug builds only, this is low risk. If any are user-reachable in release, they should be localized.

Suggested remediation:

1. Localize `Browser omnibar`.
2. Decide whether debug windows are exempt from localization.
3. Add a lightweight scan in CI for `Text("...")`, `Button("...")`, `.help("...")`, and `.accessibilityLabel("...")` outside known debug-only files/sections.

### 9. Medium-low: Focus Policy Exists, But Call Boundaries Need Guardrails

The focus-stealing policy is better than a naive implementation:

- `TerminalController` maintains a command policy stack.
- Focus-intent commands are listed explicitly (`Sources/TerminalController.swift:119-144`).
- There is a unit test asserting that `surface.trigger_flash` does not allow focus mutation (`c11Tests/TerminalControllerSocketSecurityTests.swift:113-118`).
- `AppDelegate.focusMainWindow` suppresses app activation while inside socket commands unless focus is allowed (`Sources/AppDelegate.swift:4930-4943`).

The remaining risk is that lower-level UI helpers still have unconditional activation behavior for non-socket callers:

- `TabManager.focusTab` calls `NSApp.activate(ignoringOtherApps: true)` and `makeKeyAndOrderFront` asynchronously (`Sources/TabManager.swift:3056-3083`).

This is probably correct for direct user UI flows, but it means new socket-command code must keep using the policy-aware `TerminalController`/`AppDelegate` paths and avoid calling `focusTab` directly unless the command is focus-intent.

Suggested remediation:

1. Add a short comment on `TabManager.focusTab` explaining that it is a user-focus helper and should not be called from non-focus socket commands.
2. Prefer naming policy-aware variants explicitly, for example `selectWorkspaceWithoutActivation` vs `focusWorkspaceAndActivate`.
3. Keep adding tests like the existing `surface.trigger_flash` policy test for every new socket command that changes selected workspace/surface.

### 10. Low: Some Forced Operations Are Probably Safe, But Worth Periodic Review

A quick scan found several forced unwrap/cast/try sites. Most are acceptable in context:

- static regex construction with literal patterns,
- unavailable AppKit coder initializers,
- IOSurface casts after type checks,
- debug-only helpers.

Examples:

- `Sources/SurfaceMetadataStore.swift:196` uses `try!` for a literal regex.
- `Sources/WorkspaceMetadataKeys.swift:92` uses `try!` for a literal regex and has a swiftlint comment.
- `Sources/GhosttyTerminalView.swift:6337-6340` type-checks `IOSurface` before force casting.

One style issue:

- `CLI/c11.swift:14066-14068` checks `plan.shimPath != nil` and then force unwraps it. This is safe as written but should be changed to `if let shimPath = plan.shimPath`.

Suggested remediation:

1. Avoid adding new force unwraps in command/IO paths.
2. Fix small style cases opportunistically when nearby code is touched.

## Things That Look Healthy

- The project has a clear operator/agent framing in `CLAUDE.md`, and the skill file is treated as part of the product contract.
- Socket focus policy is explicit and tested for at least one non-focus command.
- Socket file permission handling appears conservative by default: default mode is `cmuxOnly`, and non-`allowAll` modes use `0600`.
- Remote relay tests cover multi-relay health and basic v1/v2 passthrough.
- The agent-skills installer exists in both CLI and UI, and newer strings in `AgentSkillsView.swift` are well localized.
- CI pins actions by commit SHA in the main workflow.

## Recommended Fix Order

### Immediate

1. Fix C11 CLI resolution in live shells:
   - ensure `c11` and `cmux` resolve to the active bundled CLI when inside C11,
   - update welcome docs to teach `c11`.
2. Add a remote parity test for `set-agent`, `set-status`, `set-progress`, and `set-title`.
3. Add those remote command specs or document why remote CLI is intentionally limited.

### Near Term

1. Convert hot telemetry socket commands to parse off-main and mutate async.
2. Remove or wrap raw `DispatchQueue.main.sync` from legacy telemetry commands.
3. Write the canonical validation matrix and update the PR template.
4. Add the security/threat-model doc for entitlements, URL handling, and socket modes.

### Structural

1. Split `TerminalController.swift` and `CLI/c11.swift` by command family.
2. Add ownership boundaries for browser, terminal, markdown, socket, remote, and sidebar metadata modules.
3. Add CI/static checks for user-facing localization and stale command-name docs.
