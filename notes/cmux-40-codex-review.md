I reviewed the supplied patch by applying `notes/cmux-40-code-diff.patch` into a temporary tree, since this checkout is currently on `cmux-35-themes-picker`, not `cmux-40-skill-installer`. I found several major issues around clobbering, manifest trust, and unmanaged/manual installs.

## 1. Principle Compliance

- major - `SkillInstaller.install` will delete and replace an existing package directory even when that directory does not have a trusted c11mux manifest. `status` marks this as `.installedNoManifest` at `Sources/SkillInstaller.swift:296`, but install treats that state as refreshable and runs `removeItem(at:)` at `Sources/SkillInstaller.swift:398-410`. This means `cmux skill install` can overwrite a user-owned `~/.claude/skills/cmux` or `~/.codex/skills/cmux` directory. That violates the "one endorsed outgoing touch" principle because c11mux is not just installing its own managed copy; it is taking ownership of an existing skill directory by name. Proposed fix: fail closed on `.installedNoManifest` and `.schemaMismatch` unless the user passes an explicit destructive flag, and make the error say exactly which directory is user-owned.

```swift
if (st.state == .installedNoManifest || st.state == .schemaMismatch) && !force {
    throw SkillInstallerError(
        code: .destUnwritable,
        message: "\(dest.path) already exists but is not a c11mux-managed skill. Re-run with --force only if you want to replace it."
    )
}
```

- major - The non-Claude "Copy Command" path bypasses the installer policy. `AgentSkillsModel.copyManualCommandToPasteboard` builds `rsync -a "\(source.path)/" "\(destRoot)/"` at `Sources/AgentSkillsView.swift:127-130`. That copies the whole bundled `skills/` directory, including `skills/release` and `skills/MANIFEST.json`; `skills/MANIFEST.json:3-8` intentionally allowlists only `cmux`, `cmux-browser`, `cmux-markdown`, and `cmux-debug-windows`. It also writes no `.c11mux-skill.json`, so status later reports a no-manifest install instead of a current install. Proposed fix: make the copied command call the managed installer, e.g. `cmux skill install --tool codex`, or generate a command from `discoverPackages` that copies only installable packages and writes manifests consistently.

- major - The allowlist fails open. `readInstallableAllowlist` returns `nil` for missing or malformed `MANIFEST.json` at `Sources/SkillInstaller.swift:208-219`, and `discoverPackages` then installs every direct child with `SKILL.md` at `Sources/SkillInstaller.swift:192-205`. In this repo that includes `skills/release`, so a packaging mistake silently broadens what c11mux installs into user agent directories. Proposed fix: treat a malformed manifest as an error, and consider requiring the manifest for shipped builds.

## 2. Safety

- major - The remove guard is weaker than its comment. `SkillInstaller.remove` only checks that `.c11mux-skill.json` exists at `Sources/SkillInstaller.swift:465-478`; it does not decode the manifest, verify the schema, verify `packageName == pkg.name`, or verify that this is actually a c11mux-managed install. Any user-owned directory containing a file with that name is deleted. Proposed fix: decode `SkillInstallerRecord`, require the current schema and matching package name, and skip or error on malformed records instead of removing.

- major - The install clobber from section 1 is also the highest-risk safety issue. `remove` refuses no-manifest directories, but `install` removes them. Safety should be symmetric: if a directory is unsafe to uninstall, it is also unsafe to replace during install/update.

- minor - The clipboard shell command is not safely quoted. `Sources/AgentSkillsView.swift:129-130` wraps paths in double quotes but does not escape embedded `"`, `$`, backticks, or backslashes. User home and app paths usually avoid those characters, but this is still a pasteable shell command generated by c11mux. Proposed fix: either avoid shell path construction by copying `cmux skill install --tool <tool>`, or add a small shell-quote helper.

- OK — nothing to flag on package-name path traversal from `MANIFEST.json`. The manifest list is only an allowlist over scanned direct child directories, and `pkg.name` comes from `lastPathComponent` at `Sources/SkillInstaller.swift:201`, so manifest entries cannot inject `../`.

## 3. Idempotency

- major - "Installed current" trusts the manifest, not the installed content. `status` computes the source hash at `Sources/SkillInstaller.swift:280`, reads `.c11mux-skill.json`, and returns `.installedCurrent` solely when `record.sourceContentHash == sourceHash` at `Sources/SkillInstaller.swift:330-331`. If a user or failed copy modifies `~/.claude/skills/cmux/SKILL.md` but leaves the manifest, `cmux skill install` skips it at `Sources/SkillInstaller.swift:390-395`. Proposed fix: compute a destination content hash excluding `.c11mux-skill.json`, then treat current as `manifestHash == sourceHash && destHash == sourceHash`.

- minor - Permission and read errors are swallowed in status surfaces. CLI status maps `try? SkillInstaller.status(...) ?? []` at `CLI/cmux.swift:15717-15718`, and the SwiftUI model catches refresh errors by setting `packages = []` at `Sources/AgentSkillsView.swift:63-73`. A permission error becomes "no bundled packages found" or an empty row instead of an actionable error. Proposed fix: add an error state per target row and emit it in both human and JSON output.

- minor - An empty or broken source package set is treated as success. `discoverPackages` can return `[]`, and install then creates the destination root and reports success with empty `installed/refreshed/skipped` arrays. If the bundled `skills/` dir exists but contains no installable packages, that should be an error, not a successful no-op.

## 4. Hash Canonicalization Correctness

- major - The hash framing is ambiguous for arbitrary file bytes. `contentHash` appends `relativePath + NUL + fileBytes + NUL` for each file at `Sources/SkillInstaller.swift:254-262`. Because file bytes can contain NULs, two different directory shapes can produce the same byte stream without breaking SHA-256. Example shape: one file whose content embeds `NUL + nextRelativePath + NUL` can collide structurally with two files. Proposed fix: length-prefix each path and each file payload before hashing.

- major - The dotfile filter is too broad. `Sources/SkillInstaller.swift:247` skips any path starting with `.` or containing `/.`, but install copies dotfiles from source. If a skill package adds a functional hidden file later, its changes do not perturb the source hash, so installs skip stale copies. Also, the manifest being filtered out is not a real source-hash problem because `contentHash` is currently only called on source package directories. Proposed fix: include dotfiles in source hashes, or skip only `SkillInstaller.manifestFilename` when hashing a destination directory.

- OK — nothing to flag on nondeterministic enumeration order. The code gathers entries and sorts by relative path at `Sources/SkillInstaller.swift:252`. The comparison is not locale-dependent in the way Finder sorting would be.

## 5. CLI Surface

- minor - Option parsing is permissive enough to misparse missing values. `optionValue` and `hasFlag` at `CLI/cmux.swift:8829-8835` do not understand `--tool=codex`, do not reject unknown flags, and treat another flag as a value. For example, `cmux skill install --home --json` parses the home as `--json`. Proposed fix: use a small structured parser for this subcommand that rejects missing option values and leftover unknown args.

- minor - JSON key ordering is not stable. `jsonString` uses `.prettyPrinted` and `.withoutEscapingSlashes` at `CLI/cmux.swift:10516-10524`, but not `.sortedKeys`. The row order is stable, but object key order is not guaranteed. Proposed fix: add `.sortedKeys` for command output that may be consumed by scripts.

- minor - `cmux skill status` hides source/status failures. Same issue as section 3: `CLI/cmux.swift:15717-15718` collapses status errors to empty packages. JSON consumers cannot distinguish "no installable packages" from "could not read source/destination".

- OK — nothing to flag on `--home` being honored by the main status/install/remove flows. `parseSkillHomeOverride` is used by all three at `CLI/cmux.swift:15702-15704`, `CLI/cmux.swift:15776-15782`, and `CLI/cmux.swift:15848-15852`.

- OK — nothing to flag on missing `~/.codex/` for `cmux skill install --tool codex`. The detection guard at `CLI/cmux.swift:15784-15790` fails cleanly with a human error or JSON error and exit code 2.

## 6. SwiftUI / AppKit Integration

- major - The onboarding window ownership is wrong. `agentSkillsOnboardingWindow` is weak at `Sources/AppDelegate.swift:6071`, while the created `NSWindow` is only held in a local variable before assigning to that weak property at `Sources/AppDelegate.swift:6087-6102`. `isReleasedWhenClosed = false` does not replace a strong owner. Depending on AppKit retention, this can either disappear unexpectedly or leave a closed panel that is not cleaned up. Proposed fix: make the property strong, install a close handler/delegate, and nil the property on close.

- minor - Closing the titlebar close button does not update any dismissal state. The `onDismiss` closure only runs for the SwiftUI buttons at `Sources/AgentSkillsView.swift:399-413`; clicking the window close button at `Sources/AppDelegate.swift:6089` closes the panel without setting a run-level "later" flag or `sheetShownKey`. If another welcome workspace triggers `sendWelcomeCommandWhenReady`, the panel can reappear in the same run. Proposed fix: centralize dismissal in a window delegate and track `dismissedThisLaunch`.

- OK — nothing to flag on the 1.2s async dispatch as a runtime main-thread call. `DispatchQueue.main.asyncAfter` at `Sources/AppDelegate.swift:6063-6065` runs the UI presentation on the main queue. If the project moves to stricter Swift concurrency, annotate the AppDelegate/presentation methods `@MainActor` to make that contract explicit.

## 7. Localization Coverage

- major - Several user-facing Settings messages are hardcoded English. `formatInstallMessage` and `formatRemoveMessage` build strings like `installed:`, `refreshed:`, `skipped:`, `no-op`, and `nothing to remove` at `Sources/AgentSkillsView.swift:140-156`, then display them in `Text` at `Sources/AgentSkillsView.swift:177-179` and `Sources/AgentSkillsView.swift:392-394`. Proposed fix: add localized string keys for the action result fragments or build a localized sentence per result state.

- minor - Installer errors are English strings from core logic and are shown directly in SwiftUI at `Sources/AgentSkillsView.swift:97-100` and `Sources/AgentSkillsView.swift:115-118`. CLI errors can stay plain, but UI presentation should map `SkillInstallerError.Code` to localized messages and include paths as interpolated arguments.

- OK — nothing to flag on the new static labels/buttons/status chips. The added keys exist in `Resources/Localizable.xcstrings` for `agentSkills.*`, `settings.agentSkills.note`, and `settings.section.agentSkills`.

## 8. Anything Else

- minor - The Settings note says "For other agents, c11mux copies a ready-to-paste command" at `Sources/cmuxApp.swift:5832-5835`, but detected non-Claude rows also show direct `Install`/`Update` buttons at `Sources/AgentSkillsView.swift:327-353`. The behavior may be acceptable as explicit operator intent, but the copy should be aligned with the actual UI contract.

- minor - `AgentSkillsModel.refresh` performs filesystem enumeration and SHA-256 hashing on the main actor at `Sources/AgentSkillsView.swift:45-84`. The current skill payload is small, so this is not urgent, but if skills grow or live on slow storage the Settings pane will block. Proposed fix: move status computation off-main and publish the rows back on the main actor.

- nit - The patch includes `docs/c11mux-snapshot-restore-plan.md`, which is unrelated to CMUX-40 and has a trailing-whitespace warning from `git apply`. Keep it out of this commit unless the PR intentionally batches CMUX-37 planning.

Overall verdict: the feature direction fits the c11mux principle, but I would not ship this patch as-is. The managed installer needs to fail closed around existing user-owned directories, validate manifests before deleting, hash actual installed content for idempotency, and make the non-Claude manual command go through the same allowlisted/manifested path. After those fixes, the remaining issues are mostly polish: stricter CLI parsing, stable JSON key order, window ownership cleanup, and localization of action-result strings.
